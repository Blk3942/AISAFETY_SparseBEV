# 优化建议

## 

从"这个数据库未来要支撑什么分析场景"出发，我提出以下 **增强建议**：

### 2.1 📊 补充「逐类别指标」明细表

**现状**：`metrics_official` 和 `metrics_safety_critical` 只存了全局聚合指标（mAP、NDS 等）。`raw_json` 里虽然有逐类别数据，但 JSONB 查询效率低且不利于可视化工具直接接入。

**建议**：新增 `metrics_per_class` 表：

```sql
CREATE TABLE metrics_per_class (
    id          BIGSERIAL PRIMARY KEY,
    run_id      BIGINT NOT NULL REFERENCES experiment_run(run_id) ON DELETE CASCADE,
    eval_type   VARCHAR(32) NOT NULL,   -- 'official' | 'safety_critical'
    class_name  VARCHAR(64) NOT NULL,
    ap          DOUBLE PRECISION,
    ate         DOUBLE PRECISION,
    ase         DOUBLE PRECISION,
    aoe         DOUBLE PRECISION,
    ave         DOUBLE PRECISION,
    aae         DOUBLE PRECISION,
    UNIQUE (run_id, eval_type, class_name)
);
```

**价值**：
- 直接支撑 **雷达图**（各类别 AP 对比）、**热力图**（类别 × 误差类型）
- 跨 run 对比时可直接 `GROUP BY class_name` 做趋势分析
- 快速定位"哪个类别拖了 NDS 后腿"

---

### 2.2 📏 补充「逐距离段指标」明细表

**动机**：Safety-critical 评估的核心关注之一是**距离衰减**——近处目标和远处目标的检测质量差异巨大。

```sql
CREATE TABLE metrics_per_distance (
    id            BIGSERIAL PRIMARY KEY,
    run_id        BIGINT NOT NULL REFERENCES experiment_run(run_id) ON DELETE CASCADE,
    eval_type     VARCHAR(32) NOT NULL,
    class_name    VARCHAR(64) NOT NULL,
    dist_bin_lo   DOUBLE PRECISION NOT NULL,  -- e.g. 0, 10, 20, 30, 40
    dist_bin_hi   DOUBLE PRECISION NOT NULL,  -- e.g. 10, 20, 30, 40, 50
    ap            DOUBLE PRECISION,
    ate           DOUBLE PRECISION,
    num_gt        INTEGER,
    num_pred      INTEGER,
    UNIQUE (run_id, eval_type, class_name, dist_bin_lo)
);
```

**价值**：
- 绘制 **AP-距离衰减曲线**，直观展示模型"视距极限"
- Safety 场景下，可量化"30m 内行人检测率是否达标"

---

### 2.3 🏷️ 为 `experiment_run` 增加实验标签系统

**现状**：`experiment_run` 只有 `notes` 自由文本，难以结构化筛选。

**建议**：增加标签表或直接加列：

```sql
-- 方案 A：轻量级——直接加列
ALTER TABLE experiment_run ADD COLUMN backbone VARCHAR(64);      -- e.g. 'ResNet-50', 'VoVNet-99'
ALTER TABLE experiment_run ADD COLUMN input_resolution VARCHAR(32); -- e.g. '256x704'
ALTER TABLE experiment_run ADD COLUMN temporal_frames INTEGER;    -- e.g. 8
ALTER TABLE experiment_run ADD COLUMN tags JSONB DEFAULT '[]';   -- 灵活标签

-- 方案 B：多对多标签表（更灵活）
CREATE TABLE run_tag (
    run_id  BIGINT NOT NULL REFERENCES experiment_run(run_id) ON DELETE CASCADE,
    tag_key VARCHAR(64) NOT NULL,
    tag_val VARCHAR(128) NOT NULL,
    PRIMARY KEY (run_id, tag_key)
);
```

**价值**：
- 支撑 **分组对比看板**：按 backbone / 分辨率 / 时序帧数分组，对比 NDS 趋势
- BI 工具（Grafana / Superset）可直接用标签做 filter



---

### 2.4 🔗 补充 `match_pair` 表（文档 §7 提到但未正式定义）

这是混淆矩阵的**底层证据链**，对可视化和 debug 极其有用：

```sql
CREATE TABLE match_pair (
    id            BIGSERIAL PRIMARY KEY,
    run_id        BIGINT NOT NULL REFERENCES experiment_run(run_id) ON DELETE CASCADE,
    eval_type     VARCHAR(32) NOT NULL,
    dist_th_m     DOUBLE PRECISION,
    gt_box_id     BIGINT REFERENCES ground_truth_box(id),    -- NULL = FP
    pred_box_id   BIGINT REFERENCES prediction_box(id),      -- NULL = FN
    match_dist    DOUBLE PRECISION,
    gt_class      VARCHAR(64),
    pred_class    VARCHAR(64),
    is_tp         BOOLEAN NOT NULL
);
CREATE INDEX idx_match_run ON match_pair (run_id, eval_type);
```

**价值**：
- 支撑 **BEV 可视化**：在鸟瞰图上同时渲染 GT 框、预测框、匹配线
- 可追溯每个 FP/FN 的具体 3D 位置和类别，做 **failure case 分析**
- 混淆矩阵的每个 cell 可下钻到具体框

---

### 2.5 📐 预计算物化视图加速常用查询

对于高频分析场景，建议创建物化视图：

```sql
-- 跨 run 对比看板的核心视图
CREATE MATERIALIZED VIEW mv_run_summary AS
SELECT
    r.run_id,
    r.project_name,
    r.config_path,
    r.eval_split,
    r.created_at,
    mo.nd_score   AS official_nds,
    mo.mean_ap    AS official_map,
    ms.nd_score   AS safety_nds,
    ms.mean_ap    AS safety_map,
    (SELECT COUNT(*) FROM prediction_sample ps WHERE ps.run_id = r.run_id) AS num_samples,
    r.avg_latency_ms
FROM experiment_run r
LEFT JOIN metrics_official mo ON mo.run_id = r.run_id
LEFT JOIN metrics_safety_critical ms ON ms.run_id = r.run_id;
```

**价值**：BI 工具直接查视图，无需每次 JOIN 多表。
